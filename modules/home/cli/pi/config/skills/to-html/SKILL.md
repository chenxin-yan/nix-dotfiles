# to-html

Produce a **single self-contained `.html` file** in the **current working
directory** that explains, plans, reviews, or otherwise visualizes
whatever the user asked about — then open it in the default browser.

The point is reviewability: a human should be able to glance at the
rendered page and grasp the structure faster than reading the same
content in a chat transcript.

## When to use this skill

- "Make an HTML for this plan" / "render this as a page".
- "Show me a review of the changes you made" as HTML.
- The user wants to share or sit with a design before approving it.
- The content has structure that visuals make obvious (flows, before/after,
  trade-off matrices, file trees, dependency graphs, timelines).

If the user just wants prose, write prose. This skill is for when a
visual artifact actually helps.

## When the user passes context

Treat any free-form text after the invocation as the **topic** and any
**emphasis hints**. Examples:

- `/skill:to-html the auth refactor plan` → topic is the auth refactor.
- `/skill:to-html review my last 3 commits, emphasize the risky parts` →
  build a review page that visually flags the risky changes.

If the topic is ambiguous (e.g. just `/skill:to-html` with no context),
ask one short clarifying question before generating.

## Workflow

1. **Decide the slug + path.** Kebab-case the topic into a short slug
   (e.g. `auth-refactor-plan`, `pr-42-review`, `cache-design`) and
   write to `./<slug>.html` in the current working directory. If that
   file already exists, append a short ISO-ish suffix
   (`./<slug>-YYYYMMDD-HHMM.html`) rather than clobber it.
2. **Pick the visual structure** before writing markup (see "Use HTML's
   affordances" below). Decide which diagrams, tables, or comparisons
   actually carry the message.
3. **Pick the interactivity budget** (see "Make it interactive" below).
   The bigger and denser the page, the more interactivity you owe the
   reviewer. Aim to make the reader's *next action* obvious at every
   point on the page.
4. **Write one self-contained `.html` file.** No external assets beyond
   CDN scripts. Anyone should be able to double-click the file and have
   it render correctly offline-after-first-load.
5. **Open it.**
   - macOS: `open <file>`
   - Linux: `xdg-open <file>` (fall through silently if it's not
     installed)
   - Either way, always print the **absolute path** to the file so the
     user can click it from the terminal too.
6. **Summarize in chat** in 2–4 bullets: what's on the page, where it
   lives, how to regenerate. Don't repeat the page content in chat.

## File template

Start from this skeleton and add only what the content needs. Do not
ship empty sections — delete what you don't use.

```html
<!doctype html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1" />
  <title><!-- topic --></title>
  <script src="https://cdn.tailwindcss.com"></script>
  <!-- Include Mermaid only if the page actually uses a diagram. -->
  <script type="module">
    import mermaid from "https://cdn.jsdelivr.net/npm/mermaid@10/dist/mermaid.esm.min.mjs";
    mermaid.initialize({ startOnLoad: true, theme: "default" });
  </script>
  <!-- Include highlight.js only if the page actually shows code. -->
  <link rel="stylesheet"
        href="https://cdn.jsdelivr.net/gh/highlightjs/cdn-release/build/styles/github.min.css" />
  <script src="https://cdn.jsdelivr.net/gh/highlightjs/cdn-release/build/highlight.min.js"></script>
  <script>addEventListener("DOMContentLoaded", () => hljs.highlightAll());</script>
</head>
<body class="bg-slate-50 text-slate-900">
  <main class="mx-auto max-w-4xl px-6 py-10 prose prose-slate">
    <header class="not-prose mb-8">
      <p class="text-xs uppercase tracking-wider text-slate-500">Generated by pi · <!-- date --></p>
      <h1 class="text-3xl font-bold"><!-- topic title --></h1>
      <p class="text-slate-600"><!-- one-sentence summary --></p>
    </header>

    <!-- Sections go here -->
  </main>
</body>
</html>
```

Notes on the template:

- **Tailwind via CDN** is fine for this skill — these pages are
  ephemeral review artifacts, not production. Use `prose` for long
  text blocks and Tailwind utilities for layout.
- **Drop libraries you don't use.** No Mermaid block if there's no
  diagram. No highlight.js if there's no code. Keep the file small.
- **Single file only.** No sibling `.css` / `.js` / image files. If you
  need an illustrative image, inline it as a base64 data URL or draw
  it as SVG / Mermaid.

## Use HTML's affordances

This is the whole point of the skill — don't ship a page that's just a
markdown dump. Reach for these when they fit:

- **Mermaid diagrams** for flows, state machines, sequence diagrams,
  architecture sketches, decision trees. Use `<pre class="mermaid">` blocks.
- **Side-by-side comparisons** (`grid grid-cols-2`) for before/after,
  option A vs option B, current vs proposed.
- **Tables** for trade-off matrices, file-by-file change summaries, or
  any "thing × attribute" data.
- **Inline SVG** for small custom diagrams Mermaid can't express well
  (rough sketches, callouts, annotated screenshots).
- **`<details>` / `<summary>`** for "show full diff" or "show raw
  output" sections that would otherwise drown the page.
- **Syntax-highlighted code** via highlight.js (`<pre><code class="language-ts">…</code></pre>`)
  for snippets longer than a line or two.
- **Status badges / callouts** (small Tailwind pills:
  `bg-green-100 text-green-800`, `bg-amber-100 text-amber-800`,
  `bg-rose-100 text-rose-800`) to flag risk, novelty, or status.
- **Anchor links + a small ToC** when the page has 4+ sections.

If the content is genuinely one paragraph of prose, this skill is
overkill — say so and write the paragraph in chat instead.

## Make it interactive: reduce cognitive load

A static dump is barely better than chat. The whole reason for HTML is
that the reviewer can **navigate, filter, and progressively reveal**
detail at their own pace. Follow this principle: **the page should
show the gist on first paint, and let the reviewer drill into any
section in one click.**

Reach for these patterns whenever the page has more than ~2 sections
or more than ~1 screen of content:

- **Sticky in-page navigation.** A small fixed ToC (sidebar on desktop,
  top-anchored on mobile) with smooth-scroll links to each `<section
  id="…">`. Highlight the active section via `IntersectionObserver`.
- **Progressive disclosure with `<details>`.** Collapse long sections
  (full diffs, raw output, exhaustive trade-off tables, appendices) by
  default. Keep the summary text on the `<summary>` line informative
  enough that the reviewer can skip without opening.
- **Default state matters.** Important things expanded; reference
  material collapsed. Never make the reviewer click to see the answer
  to the question the page is supposedly answering.
- **In-page filter / search.** When listing many items (files,
  commits, decisions, options), add a single `<input type="search">`
  that hides non-matching cards in real time. ~15 lines of vanilla JS.
- **Tag chips for faceted filtering.** Multi-axis filtering on review
  pages (severity, file area, status) via toggleable pill buttons that
  drive `data-*` attribute matching.
- **Tabs for parallel variants.** Before/after, option A/B/C, mobile/
  desktop mockups — tabs avoid making the reviewer scroll back and
  forth to compare. Plain `<button>` + `hidden` class is enough.
- **"Mark as reviewed" checkboxes.** Per-section checkboxes whose
  state persists to `localStorage` (keyed by file slug + section id).
  Lets a reviewer pause and resume across days.
- **Theme toggle.** Respect `prefers-color-scheme` by default and
  offer a light/dark toggle. Persist via `localStorage`. Pages get
  reviewed at 11pm.
- **Keyboard shortcuts** when the page is list-heavy: `/` to focus
  search, `j`/`k` to jump between sections, `?` to show a help overlay.
  Only add when they pay for themselves.
- **Cross-references.** When section A says "see section C", make it
  an `<a href="#sec-c">` that smooth-scrolls *and* briefly highlights
  the target (e.g. add `.ring-2.ring-amber-400` for 1.5s on
  `:target`).
- **Persistent `<details>` state.** A tiny script can save
  `open`/closed state per `id` to `localStorage` so a reviewer's
  expansion choices survive a refresh.
- **Print / PDF friendly.** Add a `@media print` block that hides the
  sidebar/toggles and expands every `<details>`. Reviewers will print
  to PDF to share.

Do **not** add interactivity that doesn't earn its keep. A 3-section
page doesn't need keyboard shortcuts. A page with no list of items
doesn't need a filter input. Match the complexity of the controls to
the complexity of the content.

### Drop-in interactivity scaffold

Include this `<script>` block (or a subset) inline at the bottom of
`<body>` when the page would benefit. It's framework-free, ~40 lines,
and handles filter + persisted `<details>` + active-ToC highlight +
theme toggle.

```html
<script>
  // 1. In-page filter: any <input data-filter="<selector>"> hides
  //    elements matching the selector whose textContent doesn't match.
  for (const input of document.querySelectorAll("input[data-filter]")) {
    const targets = document.querySelectorAll(input.dataset.filter);
    input.addEventListener("input", () => {
      const q = input.value.trim().toLowerCase();
      for (const el of targets) {
        el.hidden = q && !el.textContent.toLowerCase().includes(q);
      }
    });
  }

  // 2. Persisted <details> state. Add id="..." to any <details> you
  //    want remembered across reloads.
  for (const d of document.querySelectorAll("details[id]")) {
    const key = `details:${location.pathname}:${d.id}`;
    const saved = localStorage.getItem(key);
    if (saved !== null) d.open = saved === "1";
    d.addEventListener("toggle", () =>
      localStorage.setItem(key, d.open ? "1" : "0")
    );
  }

  // 3. Persisted "reviewed" checkboxes.
  for (const cb of document.querySelectorAll("input[type=checkbox][data-reviewed]")) {
    const key = `reviewed:${location.pathname}:${cb.dataset.reviewed}`;
    cb.checked = localStorage.getItem(key) === "1";
    cb.addEventListener("change", () =>
      localStorage.setItem(key, cb.checked ? "1" : "0")
    );
  }

  // 4. Active-section highlight in the ToC. Mark ToC links with
  //    data-toc and matching href="#section-id".
  const tocLinks = [...document.querySelectorAll("a[data-toc]")];
  if (tocLinks.length) {
    const byId = new Map(tocLinks.map((a) => [a.getAttribute("href").slice(1), a]));
    const io = new IntersectionObserver(
      (entries) => {
        for (const e of entries) {
          const link = byId.get(e.target.id);
          if (!link) continue;
          link.classList.toggle("font-semibold", e.isIntersecting);
          link.classList.toggle("text-slate-900", e.isIntersecting);
        }
      },
      { rootMargin: "-40% 0px -55% 0px" }
    );
    for (const id of byId.keys()) {
      const el = document.getElementById(id);
      if (el) io.observe(el);
    }
  }

  // 5. Theme toggle. Button with id="theme-toggle" flips dark mode
  //    and persists. Defaults to prefers-color-scheme.
  const root = document.documentElement;
  const themeKey = "theme";
  const setTheme = (t) => {
    root.classList.toggle("dark", t === "dark");
    localStorage.setItem(themeKey, t);
  };
  setTheme(
    localStorage.getItem(themeKey) ??
      (matchMedia("(prefers-color-scheme: dark)").matches ? "dark" : "light")
  );
  document.getElementById("theme-toggle")?.addEventListener("click", () =>
    setTheme(root.classList.contains("dark") ? "light" : "dark")
  );
</script>
```

For Tailwind dark mode to work, also configure it on the CDN script:

```html
<script>tailwind.config = { darkMode: "class" };</script>
```

added right after the Tailwind `<script>` tag.

## Examples of good shape

- **Plan page**: header → one-sentence goal → "why" callout → Mermaid
  flow of the proposed change → file-by-file table → open questions
  list → risks callout.
- **Change-review page**: header with branch/commit → summary callout
  → per-file collapsible diffs with inline commentary → Mermaid
  before/after architecture → risks + rollback notes.
- **Concept explainer**: header → one-paragraph TL;DR → annotated
  diagram → step-by-step walkthrough with code snippets → links to
  source.

## Hard rules

- **One file only.** No sibling assets. CDN scripts are the only
  external dependency.
- **Current working directory.** Don't write into `tmp/`, `docs/`,
  `$TMPDIR`, or anywhere else unless the user explicitly asked.
- **Don't overwrite an existing file** with the same slug — suffix
  with a timestamp instead.
- **Always print the absolute path** in the chat summary, even after
  opening the browser, so the user can re-open or share it.
- **Don't claim it's open** if `open` / `xdg-open` failed — say so and
  give the path.
- **Interactivity must serve review, not decorate it.** Every control
  on the page should answer a question the reviewer would otherwise
  have to scroll to answer. If a control doesn't earn its keep, cut
  it.
